Capstone Proposal: Noemi’s Bakery Inventory & Order Taking API
1. Project Concept
Theme: Inventory & Order Management

Purpose: Noemi’s is my famiy's bakery, specializing in traditional filipino rice cakes (kakanin). This API digitizes manual stock tracking and automates the transition from Customer Order to Kitchen Prep List. By managing live inventory and using a scheduled background task to aggregate daily orders into a summarized "Bake List," the API ensures the kitchen knows exactly how many batches to prepare each morning.

2. API Resources (The Three Main Collections)
Resource 1: Products (Inventory)
The digital representation of the bakery's display case.

Fields: name, productId, currentStock, lowStockThreshold.

Endpoints:

GET /api/v1/products: (Public) Returns a list of all kakanin varieties and their current availability.

PUT /api/v1/products/:productId: (Staff Only) Updates stock levels (restocking) or modifies item details.

POST /api/v1/products: (Manager Only) Adds a new kakanin variety to the permanent menu.

DELETE /api/v1/products/:productId: (Manager Only) Removes a discontinued product from the database.

Resource 2: Orders (Platter Reservations)
Handles the complex logic for custom kakanin assortments and customer data.

Platter Logic: Supports 12, 26, 36, 50, 75, and 100-piece platters.

Constraints: Max 6 varieties per platter; Minimum 6 pieces per variety.

Endpoints:

POST /api/v1/orders: (Public) Submits a new order. Validates platter math and updates product stock.

GET /api/v1/orders: (Staff Only) Returns a list of all active orders for shop management.

GET /api/v1/orders/:id: (Public/Staff) Retrieves the status and details of a specific order slip.

PUT /api/v1/orders/:id: (Staff Only) Updates the order status (ex. "Pending" to "Ready for Pickup").

DELETE /api/v1/orders/:id: (Staff Only) Removes an order from the active queue once completed or cancelled.

Resource 3: BakeLists (Daily Summaries)
The automated output for the kitchen team.

Description: Aggregated documents generated by a scheduled task to summarize total production needs.

Endpoints:

GET /api/v1/bakelists: (Staff Only) Returns a history of generated prep lists.

GET /api/v1/bakelists/:date: (Staff Only) Fetches the specific production totals for a given date (e.g., /2026-03-30) to see total pieces required per kakanin type.

3. New Component Research: node-cron
Research Goal: I will research how to implement a server-side scheduler to automate data aggregation and reporting.

Practical Value: I will implement a task that triggers nightly at closing. This task will query the Orders collection for the following day, aggregate the total pieces required for every kakanin variety, and generate a single BakeList document. This automation eliminates the need for manual counting, reducing error in next day preperations.
